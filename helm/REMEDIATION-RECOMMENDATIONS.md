## Open Items

### CRITICAL — Resolve Before Any Production Deployment

---

#### C-03 — CORS Wildcard on OI-RAG API

**Source:** Security Audit C-03

**Status:** Deferred — will be addressed once the consumers of the RAG API are defined and the set of permitted origins is known. Fix is a one-line change (see below).

**File:** `charts/hybridsearch/values.yaml`, line 14

**Risk:** `CORS_ALLOW_ORIGINS: "*"` allows any origin to make cross-origin requests to the OI-RAG service. If the service is reachable from a browser context, an attacker's site can make authenticated cross-origin calls to the RAG pipeline.

**Recommended fix:** Restrict to the specific deployment domain:

```yaml
CORS_ALLOW_ORIGINS: "https://threesixty.objective.com"
```

### HIGH — Address Before Production

---

#### H-02a — 3Sixty Tomcat Images Require Root for WAR chmod — Non-Root Execution Blocked

**Source:** Discovered during Docker Desktop trial deployment (2026-03-16)

**Files:** `charts/threesixty/values.yaml` (`podSecurityContext`)

**Risk:** The 3Sixty Admin, Discovery, and SCIM images run a `chmod` on their WAR file at container startup. The WAR is owned by root inside the image. Enforcing `runAsUser: 1000` / `runAsNonRoot: true` causes an immediate `Operation not permitted` error and the containers crash-loop. As a result the threesixty `podSecurityContext` has been relaxed to `{}` — the containers currently run as whatever user the image specifies (root or a Tomcat user).

**Current state:** Both `podSecurityContext` and `containerSecurityContext` are set to `{}` for the threesixty subchart. The containers run with full root capabilities as the image requires. No security context hardening is active for these pods until the images are rebuilt.

**Recommended fix:** The 3Sixty images must be rebuilt to set correct file ownership at build time (e.g. `COPY --chown=1000:1000` in the Dockerfile). Once rebuilt, re-enable:

```yaml
podSecurityContext:
  runAsNonRoot: true
  runAsUser: 1000
  runAsGroup: 1000
  fsGroup: 1000
```

This requires coordination with the 3Sixty image build team.

#### H-02 — All Pods Use the Default ServiceAccount with Token Automounting

**Source:** Internal review L2 (elevated by Security Audit to HIGH)

**Risk:** No dedicated ServiceAccounts are defined. All pods share the namespace default ServiceAccount with API server tokens auto-mounted at `/var/run/secrets/kubernetes.io/serviceaccount/token`. A compromised container gets a Kubernetes API token that may have permissions to list/read secrets, pods, and other resources. Violates CIS Kubernetes Benchmark 5.1.6.

**Recommended fix:** Add a ServiceAccount template to each subchart with `automountServiceAccountToken: false`. No pod in this stack needs API server access.

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: {{ include "threesixty.fullname" . }}
automountServiceAccountToken: false
```

---

#### H-04 — Ollama Container Runs as Root (UID 0)

**Source:** Internal review M5 (elevated by Security Audit to HIGH)

**File:** `charts/hybridsearch/values.yaml`

**Risk:** `runAsUser: 0` with `runAsNonRoot` not set and no `capabilities.drop: ["ALL"]`. A container escape from Ollama would grant root access to the node. Violates Kubernetes `restricted` and `baseline` Pod Security Admission profiles.

**Stage 1 — implemented:** `capabilities.drop: ["ALL"]` added to the Ollama container security context (`charts/hybridsearch/values.yaml`). Ollama still runs as root (UID 0) but has no Linux capabilities, significantly reducing blast radius of a container escape.

**Stage 2 — pending:** Add an init container that pre-creates `/home/ollama/.ollama` owned by a non-root UID (e.g. 1000), then run Ollama with `runAsUser: 1000` and `OLLAMA_MODELS=/home/ollama/.ollama/models`. Requires testing against the specific Ollama version before deploying to production.

Monitor the [Ollama GitHub issues](https://github.com/ollama/ollama/issues) for an official non-root image as a future alternative.

---

#### H-06 — Self-Signed TLS Certificate Generated by Setup Scripts

**Source:** Security Audit H-05

**Files:** `kubectl-create-secrets.ps1` (lines ~188), `aks-create-secrets.ps1` (lines ~263)

**Status:** To be addressed by deployment personnel during production setup. See [SECURITY.md](SECURITY.md) for full instructions covering CA-signed certificates, cert-manager (HTTP-01 and DNS-01), and Azure Key Vault options.

**Risk:** Both setup scripts generate a self-signed RSA 2048-bit certificate with 365-day validity and `/CN=domain` only. Self-signed certificates provide encryption but no identity verification, enabling MITM attacks. RSA 2048 is below current best practice for new deployments.

---

#### H-07 — No Inter-Service TLS (All Internal Communication is Plaintext HTTP)

**Source:** Security Audit H-06

**Dependency:** Requires H-06 (CA-signed certificates) to be completed first — inter-service TLS is only meaningful once the cluster has a trusted CA.

**Risk:** All inter-pod communication uses `http://` URLs. MongoDB credentials, bearer tokens, and OAuth2 tokens are transmitted in plaintext. `GRPC_SERVER_SSL` is explicitly set to `"false"`.

**Actions required once H-06 is enacted:**

1. **OpenSearch TLS:** Enable the OpenSearch security plugin TLS configuration (`plugins.security.ssl.http.enabled: true`). Mount the CA cert into the OpenSearch pod and update the `OPENSEARCH_URL` env vars in OI-RAG to use `https://`.
2. **gRPC TLS:** Set `GRPC_SERVER_SSL: "true"` in the Remote Agent / Admin config (`charts/hybridsearch/values.yaml` and `charts/threesixty/values.yaml`) and provision the gRPC server certificate.
3. **MongoDB TLS:** Enable `tls` in the MongoDB connection string (`?tls=true&tlsCAFile=...`) once the MongoDB pod is configured with a certificate.
4. **Full mTLS (optional):** Deploy Istio or Linkerd to enforce automatic mTLS for all pod-to-pod communication without per-service configuration.

### MEDIUM — Address Before Wider Rollout

---

#### M-01 — No PodDisruptionBudgets

**Source:** Internal review L4 (elevated to MEDIUM by Security Audit M-02)

**Risk:** During AKS cluster node upgrades or scale-down operations, all replicas of stateful workloads (MongoDB, OpenSearch) can be evicted simultaneously causing unavoidable downtime.

**Recommended fix:** Add PodDisruptionBudget resources for MongoDB and OpenSearch with `minAvailable: 1`.

---

#### M-02 — Ollama Image Tag Pinned to `latest`

**Source:** Internal review L5 (confirmed as MEDIUM by Security Audit M-01)

**File:** `charts/hybridsearch/values.yaml`

**Risk:** `latest` tag is mutable. Different nodes can pull different versions. Combined with `pullPolicy: IfNotPresent`, new nodes will pull whatever `latest` points to at scheduling time. No rollback to a known state is possible.

**Recommended fix:** Pin to a specific semantic version (e.g., `0.6.0`). Update deliberately when testing new Ollama versions.

---

#### M-06 — MongoDB Init Script Shell Variable Expansion (Injection Risk)

**Source:** Security Audit M-05

**File:** `charts/mongo/templates/init-configmap-mongo.yaml`

**Risk:** The init script uses unquoted `${MONGO_INITDB_ROOT_USERNAME}` and `${MONGO_INITDB_ROOT_PASSWORD}` in a `mongosh` heredoc. If credentials contain shell metacharacters (backticks, `$(...)`, double quotes), they could cause shell injection during init.

**Recommended fix:** Quote the variables or use `mongosh --eval` with proper escaping instead of a heredoc.

---

#### M-07 — readOnlyRootFilesystem Not Set on Any Container

**Source:** Security Audit M-06

**Risk:** All containers have writable root filesystems, allowing an attacker who gains code execution to modify binaries, install tools, or tamper with the application.

**Recommended fix:** Add `readOnlyRootFilesystem: true` to all container security contexts and mount writable `emptyDir` volumes only where the application needs write access (`/tmp`, log directories).

---

#### M-08 — OpenSearch vm.max_map_count Not Enforced at Deploy Time

**Source:** Internal review L9

**File:** `charts/opensearch/values.yaml` (comment at lines 47–49)

**Risk:** OpenSearch will fail to start with a bootstrap check error if the host node has `vm.max_map_count < 262144`. This is a common first-deployment failure that is difficult to diagnose from pod logs alone.

**Recommended fix (choose one):**

1. Add a privileged DaemonSet that runs `sysctl -w vm.max_map_count=262144` on each node at startup
2. Add an init container to the OpenSearch deployment that checks the value and fails fast with a clear error message if too low
3. Add a Helm pre-install hook that verifies node configuration before chart installation

### LOW — Best Practice Improvements

---

#### L-01 — RC Image Tags Used in Default Chart Values

**Source:** Security Audit L-01

**Files:** `charts/threesixty/values.yaml` (tag: 5.0.3-RC1), `charts/hybridsearch/values.yaml` (tags: 5.0.1-RC3, 5.0.2-RC1)

**Risk:** Release candidate images in default values are deployed if no production values override is provided. RC images may contain known bugs or incomplete features.

**Recommended fix:** Ensure all default image tags are GA (General Availability) releases before production deployment. Confirm with the 3Sixty team which image tags are production-ready.

---

#### L-02 — OPENAI_API_KEY Placeholder in ConfigMap

**Source:** Security Audit L-02

**File:** `charts/hybridsearch/values.yaml`, line 17

**Risk:** `OPENAI_API_KEY: "no-key"` is deployed as plaintext in a ConfigMap. If an actual API key is ever needed, this pattern would leak it.

**Recommended fix:** Document explicitly that this is intentionally non-functional for the Ollama-only configuration. If ever replaced with a real key, move it to a Secret.

---

#### L-03 — ECR Token Expires After 12 Hours — No Automated Rotation

**Source:** Security Audit L-04

**Files:** `kubectl-create-secrets.ps1`, `aks-create-secrets.ps1`

**Risk:** The ECR docker-registry secret uses a token from `aws ecr get-login-password` which expires after 12 hours. After expiry, new pod scheduling will fail if the node needs to pull a new image.

**Recommended fix:** Deploy an ECR credential helper (e.g., `ecr-credential-provider` kubelet plugin or a CronJob that refreshes the secret on a schedule).

---

#### L-04 — No HTTPS Redirect on Ingress

**Source:** Security Audit L-06

**File:** `charts/ingress/templates/ingress.yaml`

**Risk:** The ingress defines TLS but does not force HTTP-to-HTTPS redirect. Users accessing via plain HTTP would transmit credentials in plaintext.

**Recommended fix:** Add Traefik middleware annotation to enforce redirect:

```yaml
annotations:
  traefik.ingress.kubernetes.io/router.middlewares: "threesixty-redirect-to-https@kubernetescrd"
```

---

#### L-05 — No Helm Chart Provenance / Signing

**Source:** Security Audit L-03

**Risk:** No `.prov` file or chart signing. Anyone with repository write access can modify chart templates without cryptographic detection.

**Recommended fix:** Consider signing Helm charts with `helm package --sign` for production deployments.

---

#### L-06 — Single-Replica Deployments for All Services

**Source:** Security Audit L-05

**Risk:** All subcharts specify `replicas: 1`. A single pod failure causes complete outage for that component. Acceptable for initial deployment but should be planned before production load.

**Recommended fix:** Plan multi-replica deployments for stateless services (admin, discovery, scim, oirag) and evaluate clustering for stateful services (MongoDB replica set, OpenSearch multi-node). Add HorizontalPodAutoscalers for stateless components once resource limits are validated through load testing.

### Overall Assessment

The security audit is competent and methodical. It correctly identifies the most consequential issues, avoids padding the finding count with noise, and the descriptions are technically accurate. The REMEDIATION-RECOMMENDATIONS.md document consolidates findings from three review passes cleanly and adds useful AKS-specific context.

There are meaningful gaps, some severity calibration issues, two factual errors verifiable from the templates, and one significant AKS-specific finding absent from both documents.

---

### Factual Corrections

**C-01 — Audit overstates history exposure.** The audit states credentials are "visible in the git history" and recommends `git filter-branch` / BFG Repo-Cleaner. The REMEDIATION document correctly clarifies that `git log --all -- values-production.yaml` returned empty — the file was never committed. The correct fix is `git rm --cached` followed by a commit. Do not run destructive history rewriting. The CRITICAL severity is still appropriate because a single accidental `git add -A && git commit` would permanently expose all credentials.

**C-02 — Audit count is off by one.** The audit lists four Helm-managed secrets (`admin`, `discovery`, `scim`, `mongodb`). `secret-discovery.yaml` is now a comment-only stub — OAuth2 was migrated to an external secret this sprint. Only three Helm-managed secrets remain. The REMEDIATION document implicitly corrects this.

---

### Severity Recalibrations

**Too low — L-04 (ECR Token Expiry): should be MEDIUM, Short-Term (not LOW, 90 days).** AKS node auto-upgrades (enabled by default) can trigger node recycling at any time, requiring a fresh image pull. A 12-hour ECR token set once at deployment will have been expired for the vast majority of the cluster's uptime. Image pull failures on rescheduled pods will surface at the worst possible moment — during an upgrade or incident response.

**Too low — L-06 (No HTTPS Redirect): should be MEDIUM, Week 1 (not LOW, 30 days).** Users access the system from day one. A Traefik middleware redirect is a single annotation change; the fix is already written out verbatim in this document. There is no justification for leaving credentials exposed over plain HTTP for 30 days.

**H-09 (Credential Duplication) as HIGH is defensible but MEDIUM is arguable.** The consequence of credential drift is service outage, not a security breach. This finding is acceptable at HIGH only because it is a direct consequence of C-02 — completing C-02 resolves it automatically.

---

### Missing Findings

**Missing A — storageClass `local-path` is a deployment blocker on AKS (MEDIUM).** The MongoDB, OpenSearch, and Elasticsearch values files reference `storageClass: local-path`. This is a k3s-specific provisioner that does not exist on AKS. A deployment from `values-production.yaml.template` without overriding `storageClass` will result in all PVCs stuck in `Pending` indefinitely — nothing starts. The correct AKS storage class is `managed-csi` (or `default`). **This is the most immediate unblocking fix required before any AKS deployment attempt.** Neither document lists this.

**Missing B — No Pod Security Admission labels on the namespace (AKS-specific).** AKS 1.25+ enforces Pod Security Admission but applies no profile to user namespaces by default. Without a `pod-security.kubernetes.io/enforce` label on `threesixty`, Ollama's root container and the unhardened Elasticsearch container can run without Kubernetes blocking them. Recommended fix: apply `enforce: baseline` to the namespace at initial deployment, with a documented `privileged` exception scoped to the Ollama pod only. Zero cost; native to AKS.

**Missing C — Azure Workload Identity / Managed Identity integration absent from roadmap (AKS-specific).** AKS supports Azure Workload Identity, which allows pods to authenticate to Azure services using a Kubernetes ServiceAccount bound to a Managed Identity — no static credentials required. For AKS, this is the preferred path over generic External Secrets Operator:

- `az aks update --attach-acr` eliminates the ECR 12-hour token expiry problem entirely (resolves L-04)
- Secrets Store CSI Driver + Azure Key Vault resolves C-02 without third-party ESO
- Workload Identity addresses H-02 (service account token exposure)

All three of these use tooling the team likely already has access to. This should appear in the Long-Term roadmap as the preferred AKS path.

**Missing D — gRPC endpoint authentication not documented.** `GRPC_SERVER_START: "true"` exposes gRPC on port 50052. Neither document asks whether this endpoint requires authentication. If it is unauthenticated and reachable from any pod in the namespace, this could escalate to HIGH or CRITICAL depending on what the gRPC API can do. Must be investigated and documented before production.

**Missing E — Ollama deployment args shell injection (LOW).** `deployment-ollama.yaml` templates the model name directly into `bash -c "... ollama pull {{ .Values.ollama.model }} ..."`. If the model name contains shell metacharacters, arbitrary commands execute at pod startup. LOW risk given operator-controlled values, but the same class of issue as M-06 (MongoDB init injection) and should be flagged for consistency.

**Missing F — ConfigMap exposes all OI-RAG env config to any pod.** `configmap-oirag.yaml` iterates `range .Values.oirag.env` and writes every key-value — including `OPENSEARCH_USER: "admin"` and `CORS_ALLOW_ORIGINS: "*"` — as plaintext ConfigMap data readable by any pod. Reinforces H-01 and H-02 but is not a discrete finding.

### Technical Corrections to Recommended Fixes

**H-06 (cert-manager ACME) — needs AKS qualification.** ACME HTTP-01 challenge requires the ACME server to reach the cluster ingress on port 80 from the internet. If this is an internal-only AKS deployment (ClusterIP-everything suggests it may be), HTTP-01 will fail. The correct approach for internal AKS deployments is DNS-01 challenge via Azure DNS, or Azure App Routing addon with Azure Key Vault certificate integration.

**H-07 (service mesh for mTLS) — disproportionate for an IT ops first deployment.** Istio and Linkerd add significant operational overhead. A more practical interim path: enable TLS on OpenSearch (configuration flag in the security plugin), enable gRPC TLS (`GRPC_SERVER_SSL: "true"`), and defer service mesh to a later maturity phase. Azure Service Mesh (`asm` addon) reduces burden somewhat but still requires expertise to operate safely.

**M-01 (PodDisruptionBudgets) — single-replica pre-condition must be documented.** `minAvailable: 1` on a single-replica stateful service means the pod can never be voluntarily evicted. AKS node upgrades drain nodes — they will block indefinitely waiting for the pod to reschedule. PDBs with `minAvailable: 1` only work safely with at least 2 replicas. Replica counts must be increased before PDBs are added for MongoDB and OpenSearch, or planned maintenance requires manually suspending the PDB. **This dependency must be called out in the implementation instructions.**

**L-04 (ECR rotation CronJob) — migration to ACR is better.** A refresh CronJob has a race condition — if it fails silently for one cycle and the token expires, the cluster is unprotected. The better long-term fix for AKS is `az aks update --attach-acr`, which eliminates token expiry entirely using Managed Identity.

---

### AKS-Specific Gaps Not in Either Document

**MetalLB config file risk.** `metallb-config.yaml` exists in the repository root with a note against using it on AKS. Its presence creates a risk that a new operator applies it, conflicting with AKS's native Azure Load Balancer controller. The file should be removed or archived outside the deployment directory.

**Traefik is not a built-in AKS addon.** AKS does not ship Traefik. It must be separately deployed, and its configuration (public vs internal LoadBalancer IP, TLS termination, middleware CRDs) directly affects the real-world impact of H-06 (cert-manager), L-06 (HTTPS redirect), and M-03 (release name constraint). The Traefik deployment must be validated as a prerequisite before several security recommendations can take effect.
